# Living Spec Template

This is the output format for the spec generated by /vs-core-rfc Phase 5. The coordinator reads this and produces the spec document.

## Template

```markdown
# Spec: [Feature Name]

**Status**: Proposed

## Problem Statement
[What problem this solves and why it matters. 2-4 sentences.]

## Design Decisions
[ADR-style records for each key decision]

### Decision 1: [title]
**Context:** [forces at play]
**Decision:** [what we chose]
**Alternatives considered:** [what we rejected and why]
**Consequences:** [what follows -- good and bad]

## Assumptions
[Numbered. Each assumption lists which design decisions and slices depend on it.]

A1. [Assumption] -- depended on by: [Decision N, Slice M]
A2. [Assumption] -- depended on by: [Decision N, Slice M]

## Vertical Slices
[Ordered by dependency. Each slice is independently implementable and testable.]

### Slice 1: [name]
**Purpose**: [one sentence -- what behavior this adds]
**Acceptance criteria**: [Given X, when Y, then Z -- durable language, no file paths]
**Validates assumptions**: [which A# this slice will confirm or contradict]
**Testing strategy**: [how to verify this slice works]

### Slice 2: [name]
...

## Out of Scope
[What this spec deliberately does not address]

## Open Questions
[Anything unresolved that implementation may need to answer]

## Evolution Log
[Initially empty. Updated by /vs-core-implement as assumptions are validated or invalidated.]
```

## Principles

- **Durable language**: Describe behaviors and contracts, not file paths or line numbers. An agent picking this up weeks later should still be able to execute it.
- **Spec the WHAT and WHY, never the HOW**: No implementation detail. No function signatures, no library choices, no directory structure. The agent chooses the implementation path.
- **Every assumption is numbered**: Assumptions are the backward-path triggers. When /vs-core-implement discovers one is wrong, the number traces back to which design decisions and slices are affected.
- **The Evolution Log is the backward-path mechanism**: /vs-core-implement writes to it as assumptions are confirmed, contradicted, or adjusted. This is how the spec stays alive.
- **Status tracks validation**: "Proposed" until implementation begins. "Validated" after all slices complete and the evolution log shows the design survived contact with reality.
